================================================================================
CARBONNEXT INTEGRATION — INFORMATION REQUEST
What OneCapture (ERP Sync) needs from the CarbonNext side to wire the real post
================================================================================

Prepared for: the CarbonNext programmer
Purpose:      collect everything needed to replace the stubbed "post to
              CarbonNext" step (FR-S6) with a real, working integration.
Context:      OneCapture already imports, classifies, dedups, stages and
              RELEASES emission batches into its own ledger with a canonical
              batch hash + audit trail. The only unbuilt piece is the actual
              hand-off of a released batch to CarbonNext. The client, config
              slots (api url / service token) and the destination-id column
              (client.carbonnext_company_id) already exist on the OneCapture
              side.


--------------------------------------------------------------------------------
0. GATING DECISION (answer this first — it changes the rest of the list)
--------------------------------------------------------------------------------
How should OneCapture deliver a released batch to CarbonNext?

    [ ] HTTP ingestion API   (recommended — keeps the systems decoupled and
                              matches the "company_dataentry" service identity
                              + shared service token the code already assumes)
    [ ] Direct database write (possible since the two systems share the same
                              DB stack/version, but tightly couples them)

Everything below assumes the HTTP API path. If a direct DB write is chosen,
replace "endpoint / payload" items with "target table(s) + column mapping".

[ ] How the token is passed (e.g. Authorization: Bearer <token>).

3. INGESTION PAYLOAD SCHEMA  (most important)
--------------------------------------------------------------------------------
[ ] The exact endpoint: URL + HTTP method.
[ ] The full field list + types expected PER emission line.
[X] KEY QUESTION — ANSWERED (owner decision, 2026-07-07):
    CarbonNext COMPUTES the CO2e. OneCapture does NOT send pre-computed
    tonnage — it sends the raw transaction data (category, quantity, UoM,
    amount, currency, vendor, date) for every line flagged carbon-related.
    OneCapture's job is only to differentiate carbon-related vs not.
    -> Please confirm the exact raw-data field list CarbonNext expects.

    PROPOSED PER-LINE PAYLOAD (field contract v1 — confirm/adjust):
      carbon_ref, idempotency_key, direction (forward|reversal),
      category_name, expense_type,
      quantity, unit (L|kWh|m3|km|kg),
      amount_gross, currency, net_amount, tax_amount, base_amount_myr,
      vendor, doc_date (ISO 8601), doc_no, doc_gross_total,
      cost_centre, department,
      batch: carbonnext_company_id, batch_hash, source_channel
             (eclaim | erpsync | ap)

[ ] FROM THE UI-STRUCTURE SPECS (Yan Ling, reviewed 2026-07-07) — new asks:
      [ ] SITES MASTER: every record keys to a Site (your `sites` table).
          Give us a pull/sync API (list of sites per company) so our capture
          UI can offer the same dropdown; we will forward site_id per line.
      [ ] FX RATES: all Scope-3 spend fields are "(MYR)". We will send
          base_amount_myr (human-verified gross x fx_rate). QUESTION: do you
          own an exchange-rate table we should pull (single source of truth),
          or do you accept our converted figure as-is? Both systems must use
          the SAME rate or reconciliation breaks.
      [ ] DATA SOURCE / DATA TYPE vocabularies (your `data_sources` /
          `data_types` tables): send us the allowed values so we can map our
          provenance (receipt / supplier invoice / ERP import) onto them.
      [ ] BUSINESS TRAVEL (Cat 6): we will forward employee ref + name,
          department, travel purpose, and fuel L / km / spend MYR per line —
          confirm these field names/shapes.
      [ ] Non-transactional sources (Cat 7 commuting, 8/13 leased assets,
          10/11 sold products, 14 franchises, 15 investments) are entered in
          CarbonNext directly — confirm OneCapture is out of scope for them.

[ ] PARTIAL DOCUMENTS (owner decision 2026-07-07 — confirm CarbonNext side):
    OneCapture forwards PER LINE, never per document: a bill mixing carbon
    and non-carbon lines posts only its carbon lines, so the amount received
    by CarbonNext will be LESS than the document total. This is by design
    (reconcile by reference, not totals). Please confirm:
      [ ] Line-level ingestion is supported; partial documents are acceptable.
      [ ] You will store/display the parent-document context we send with
          each line: doc_no + document gross total + currency (so a user in
          CarbonNext can see "RM 500 carbon lines of a RM 2,000 invoice").
      [ ] Spend basis: do spend-based factors apply to NET (ex-tax) or GROSS
          amounts? We can send both; tell us which you use.
      [ ] Reversals: corrections arrive as negative-amount records marked
          direction='reversal' (never edits of prior records) — supported?
      [ ] Do you dedup on your side across our channels (e-Claim / ERP Sync /
          AP invoices), or do you rely fully on our idempotency keys?
[ ] Units & conventions:
        - tonnes vs kg
        - decimal precision / rounding rules
        - scope encoding: 1/2/3  vs  "scope_1"/"scope_2"/"scope_3_*"
        - date format
[ ] Must our factor keys / factor versions match CarbonNext's own registry?
    (Ours are string-versioned, e.g. "MY-2026.1"; the shared ledger uses ints.)
[ ] Any required batch-level header fields (company id, batch id, hash, count,
    period, source system, etc.).

SEPARATE DEPENDENCY (not from the CarbonNext programmer — flagged so it is not

--------------------------------------------------------------------------------
FR-S6 also requires an RFC 3161 TSA (Time Stamp Authority) to anchor the batch
hash BEFORE posting. This is currently stubbed too and must be sourced
separately from a TSA provider — it is not part of the CarbonNext hand-off, but
it is part of the same release gate.

SUMMARY — the minimum needed to start wiring

2. Service token + how to send it                  [Section 1]
3. carbonnext_company_id for each client           [Section 2]

5. Idempotency key + ack/response format           [Section 4]

Items 5-7 (reversals, environments, reconciliation) can follow, but the five
above are enough to build and test the first real post.
